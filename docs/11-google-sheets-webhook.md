# Google Sheets Webhook

## Purpose

Google Sheets is the lightweight operations dashboard for COD orders. The
backend stays the source of truth, but every successful order is also sent to
a Google Sheet so the team can run confirmation calls and fulfillment from a
single tab.

## Sheet shape

A single tab named **`orders`** with this exact header row:

| date | orderid | country | name | phone | product | sku | quantity | total price | currency | status |
|------|---------|---------|------|-------|---------|-----|----------|-------------|----------|--------|

The Apps Script in `scripts/google-sheets/Code.gs` creates the tab and the
header row automatically on the first request.

### Column meaning

- `date` — DD/MM/YYYY in Morocco time.
- `orderid` — `HNINA-YYYYMMDD-XXXX` generated by the backend.
- `country` — always `MOROCCO`.
- `name` — customer full name as entered.
- `phone` — local Moroccan format (`06XXXXXXXX` / `07XXXXXXXX`). Stored as
  text so the leading zero is preserved.
- `product` / `sku` / `quantity` — slash-separated, one slot per ordered item,
  positionally aligned across the three columns. Example for an order with
  two products:
  - `product`  → `حنينة ماما/حنينة جذور`
  - `sku`      → `HNM-7281/HNJ-3469`
  - `quantity` → `2/1`
- `total price` — total amount in MAD (number).
- `currency` — always `MAD`.
- `status` — left empty by the backend. The ops team fills this in
  (`called`, `confirmed`, `shipped`, `delivered`, `cancelled`, …).

## Apps Script deployment

No shared secret is used. The web app is published with `Anyone` access; only
the backend ever knows the URL.

1. Open the spreadsheet → Extensions → Apps Script.
2. Paste the contents of `scripts/google-sheets/Code.gs`. Save.
3. Deploy → New deployment → Web app.
   - Execute as: **Me**
   - Who has access: **Anyone**
4. Copy the `/exec` URL.
5. Backend `.env` (or panel env vars):
   ```
   SHEET_WEBHOOK_URL=https://script.google.com/macros/s/XXXXXXXX/exec
   ```
6. (Optional) Run `testWebhook` from the Apps Script editor to drop a fake
   row and verify the wiring end-to-end.

## Backend request format

Backend POSTs JSON to `SHEET_WEBHOOK_URL`:

```json
{
  "date": "11/06/2026",
  "orderId": "HNINA-20260611-0001",
  "country": "MOROCCO",
  "name": "فاطمة الزهراء",
  "phone": "0612345678",
  "product": "حنينة ماما/حنينة جذور",
  "sku": "HNM-7281/HNJ-3469",
  "quantity": "2/1",
  "totalPrice": 597,
  "currency": "MAD",
  "status": ""
}
```

## Response contract

The Apps Script always returns HTTP 200 (Apps Script web apps cannot return
other status codes), so success is signalled in the JSON body:

```json
// success
{ "success": true, "orderId": "HNINA-20260611-0001" }
// failure
{ "success": false, "error": "no_active_spreadsheet" }
```

The backend (`app/services/sheet_webhook.py`) requires `success === true` and
treats every other response — including HTML login pages from an
incorrectly-configured deployment — as a failure. Do not change this contract
on either side without updating both.

## Failure handling

If the Sheet webhook fails:

- the customer still gets a successful order response,
- the order row in Postgres is flagged `sheet_status=failed` with the error
  body in the logs,
- it can be re-sent manually by calling
  `send_to_google_sheets(order, items)` from a one-off script.

Common `sheet_status=failed` errors and what they mean:

| Error in logs | Likely cause | Fix |
|---|---|---|
| `not_configured` | `SHEET_WEBHOOK_URL` env var is empty | Paste the `/exec` URL into the backend env and redeploy. |
| `non_json_response_likely_auth_redirect` | Deployment is **not** "Anyone" access, or the URL is `/dev` instead of `/exec` | Redeploy with "Who has access: Anyone" and use the `/exec` URL. |
| `apps_script_error: no_active_spreadsheet` | The script is a standalone Apps Script project, not container-bound to the spreadsheet | Open the spreadsheet → Extensions → Apps Script, paste `Code.gs` there, redeploy. |
| `apps_script_error: ...` (other) | Runtime error inside Apps Script (e.g. permissions, header drift) | Open Apps Script → Executions to read the stack trace. |
| `http_4xx/5xx` | Network/transport issue to `script.google.com` | Retry; check backend egress. |

> After editing `Code.gs` you must **Deploy → Manage deployments → ✎ → New
> version** for the live `/exec` URL to pick up the new code. The URL itself
> stays the same; no backend env change is needed.
